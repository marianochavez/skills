# Issue tracker: Obsidian (local vault)

Issues, PRDs, `CONTEXT.md`, and ADRs for this repo live as markdown files inside a personal Obsidian vault elsewhere on disk. Nothing is written to the repo itself — this backend is designed for work repos where you can't commit agent-generated docs or use the repo's native issue tracker.

## Where the binding lives

The vault path is **not** stored in the repo. It's stored in the user's personal config:

```
~/.config/matt-pocock-skills/repos.toml
```

Schema, one section per repo, keyed by the absolute repo root:

```toml
["/Users/<user>/work/medra-backend"]
backend = "obsidian"
vault = "/Users/<user>/Documents/MyVault"
repo_root_in_vault = "Work/Issues/medra-backend"
domain_docs_in_vault = true   # if false, CONTEXT.md and ADRs live in the repo
```

Skills resolve the binding by:

1. `git rev-parse --show-toplevel` → absolute repo root.
2. Read `~/.config/matt-pocock-skills/repos.toml`. Find the section whose key matches the repo root.
3. If no entry exists, tell the user to run `/setup-matt-pocock-skills`.

## Folder structure inside the vault

```
<vault>/<repo_root_in_vault>/
├── CONTEXT.md
├── ADRs/
│   ├── 0001-use-postgres.md
│   └── 0002-event-sourced-orders.md
├── PRDs/
│   └── <feature-slug>.md
├── Issues/
│   ├── 0001-<slug>.md
│   └── 0042-fix-login-race.md
└── _index.md
```

- `CONTEXT.md` and `ADRs/` are present only when `domain_docs_in_vault = true`.
- `Issues/` filenames are zero-padded 4-digit global IDs per repo, plus a kebab-case slug derived from the title. `0042-fix-login-race.md`.
- `_index.md` is generated by the setup with pre-built Dataview blocks for navigation. Optional — the skill never relies on it being current.

## Frontmatter schema (issues)

```yaml
---
status: needs-triage             # one of the triage roles
type: issue                      # issue | bug | enhancement
repo: medra-backend              # repo slug, for cross-repo queries
prd: "[[../PRDs/auth-refactor]]" # wikilink to the parent PRD (optional)
id: 42                           # global per-repo, matches the filename prefix
tags: [claude/issue, repo/medra-backend, status/needs-triage]
created: 2026-05-12
---
```

- `status:` is the canonical state. The skill mutates this field on every transition.
- `tags:` mirrors the state via `status/<role>`. The skill keeps the two in sync — never mutate one without the other.
- `prd:` is an Obsidian wikilink. The PRD file lists its issues with reciprocal wikilinks under an `## Issues` section: `- [[../Issues/0042-fix-login-race|#42 Fix login race]]`.
- The H1 of the file is the title — no `title:` field.

PRDs use a similar frontmatter without `prd:` and with `type: prd`.

## When a skill says "publish to the issue tracker"

Create a new file under `<vault>/<repo_root_in_vault>/Issues/` (or `PRDs/` for PRDs).

To pick the next ID:

```bash
# Count both regular and iCloud-placeholder files to be safe.
ls Issues/*.md Issues/.*.md.icloud 2>/dev/null | wc -l
```

The next ID is that count + 1. Zero-pad to 4 digits.

## When a skill says "fetch the relevant ticket"

The user typically passes a reference like `#42`, `42`, or a slug. Resolve it as:

```bash
ls "<vault>/<repo_root_in_vault>/Issues/0042-"*.md
```

Read that file. The frontmatter and body together are the ticket.

## When a skill says "comment on the ticket"

Append to the file under a `## Comments` heading. Each comment starts with a date in ISO-8601 (`2026-05-12`) followed by the body. Don't rewrite earlier comments.

## When a skill says "apply / remove a triage label"

Edit the frontmatter `status:` field and the corresponding `status/<role>` tag in `tags:` — keep them in sync. Don't add the same `status/*` tag twice if a previous run left one behind.

## Listing issues by state

Use `ripgrep` over the issues folder; don't depend on Obsidian being open or any plugin being installed:

```bash
rg --type md -l '^status: needs-triage$' "<vault>/<repo_root_in_vault>/Issues/"
```

For richer queries (counts, group-by, etc.), the setup pre-populates `_index.md` with Dataview blocks for the user to browse in Obsidian. The skill itself never executes Dataview — its source of truth is the frontmatter, read with `rg`.

## Constraints

- **Single-machine assumption.** If the vault is synced across machines (iCloud, Obsidian Sync, Dropbox), avoid creating issues offline from two machines simultaneously — ID collisions are possible. The skill counts `.icloud` placeholders defensively but does not resolve sync conflicts.
- **Obsidian does not need to be running** for any operation. Files are read and written directly. Obsidian picks up changes the next time it focuses the vault.
- **Never write outside the configured `repo_root_in_vault`.** Even if the user moves files around in Obsidian, the skill writes only inside the configured subfolder.
